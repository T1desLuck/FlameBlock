# FlameBlock — техническая спецификация (v1.0.0)

Полный инвентарь того, что уже реализовано, и как каждая часть работает. Это база, от
которой отталкивается RESEARCH.md — документ с разбором того, что можно добавить дальше.

## 1. Архитектура целиком

```
Сайт грузится
   │
   ├─► Сетевые запросы ──► declarativeNetRequest (движок браузера, не наш код)
   │                         сверяет каждый запрос с rules/{ads,trackers,antiadblock}.json
   │                         и либо блокирует его, либо пропускает — ДО того, как наш
   │                         JS вообще успевает что-то увидеть
   │
   └─► DOM страницы ──► content.js (ISOLATED world, document_start)
                          инъецирует <style> со списком селекторов, прячет совпадения,
                          следит за MutationObserver за новыми узлами, шлёт счётчик
                          в background.js через runtime.sendMessage

background.js (service worker, живёт только пока нужен, может перезапускаться)
   ├─ считает заблокированное per-tab через chrome.storage.session (не через
   │  обычную переменную — service worker не персистентный, обычный массив/Map
   │  исчезнет при перезапуске)
   ├─ рисует бейдж на иконке (chrome.action.setBadgeText/BackgroundColor)
   ├─ управляет вайт-листом сайтов через динамические правила DNR (allow-правило
   │  с приоритетом 2, перебивает базовые block-правила с приоритетом 1)
   ├─ управляет своими правилами пользователя (приоритет 3, перебивает всё)
   └─ отвечает на сообщения от popup/options/content.js

popup/ и options/ — обычные html/css/js, общаются с background.js через
chrome.runtime.sendMessage, ничего не решают сами.
```

Ключевая идея: почти вся тяжёлая работа (сравнение каждого сетевого запроса со
списками) выполняется НЕ нашим JS-кодом, а нативным движком declarativeNetRequest
внутри самого браузера. Наш код почти не участвует в горячем пути — отсюда и низкая
нагрузка на систему, о которой просили изначально.

## 2. Файлы и их роль

| Файл | Что делает |
|---|---|
| `manifest.json` | Manifest V3, единый для Chrome и Opera. Права: `declarativeNetRequest`, `storage`, `webRequest`, `webNavigation`, `tabs`, host-доступ `<all_urls>`. |
| `background.js` (220 строк) | Service worker: бейдж, per-tab счётчик через `storage.session`, управление вайт-листом и своими правилами через DNR, обработка сообщений. |
| `content.js` (113 строк) | Косметическая фильтрация в изолированном мире страницы: инъекция `<style>`, MutationObserver, подсчёт скрытого. |
| `common.js` (54 строки) | Общие настройки по умолчанию и хелперы, подключается и в background, и в popup/options. |
| `popup/` | Всплывающее окно: тумблер «реклама на этом сайте», счётчики, пауза, переход в настройки. |
| `options/` | Расширенные настройки: категории вкл/выкл, вайт-лист с удалением, свои правила, статистика, блок «о разработчике». |
| `icons/` | 16/32/48/128px, сделаны из загруженного лого. |
| `generate_rules.py` | Скрипт, которым собраны все `rules/*.json` — можно перезапустить после обновления исходных списков. |

## 3. Списки блокировки — откуда взяты и что покрывают

Все три файла — не выдуманные списки, а результат разбора настоящих
community-списков **EasyList / EasyPrivacy** (официальный репозиторий
`github.com/easylist/easylist`), плюс вручную проверенный список крупных сетей.
Парсер (`generate_rules.py`) намеренно берёт только простые правила вида
`||domain^` (и `$third-party`/`$important`/`$popup`) — всё сложное (косmetические
правила `##`, regex, `domain=...`) пропускается, чтобы исключить риск одной
битой строки, ломающей загрузку всего набора.

| Файл | Правил | Источник |
|---|---|---|
| `rules/ads.json` | 21 148 | curated-список + `easylist_adservers*` + точечно проверенные домены Яндекс/Mail.ru/VK/Unity Ads + срез `RU AdList` (github.com/easylist/ruadlist, официальный региональный список поверх EasyList, до 4000 правил) |
| `rules/trackers.json` | 4 800 | curated-список + `easyprivacy_*` + метрика/счётчики Яндекса и Mail.ru из RU AdList |
| `rules/antiadblock.json` | 2 313 | curated-список + список слежения Admiral из EasyPrivacy + срез Adblock Warning Removal List из RU AdList |

**v1.2.1:** EasyList/EasyPrivacy — списки в основном западные и слабо покрывают рекламу
Яндекса/Mail.ru/VK: эти сети используют инфраструктуру, разделяемую с обычными
(не рекламными) сервисами, поэтому в RU AdList они по большей части закрываются не
блокировкой домена целиком, а точечными CSS-правилами. Патч (`patch_ru_networks.py`)
добавил проверенные по нескольким независимым источникам домены (bs.yandex.ru,
awaps.yandex.ru и т.д.), плюс общий срез RU AdList, плюс Unity Ads и соседние
мобильные/WebGL-сети, плюс недостающий CSS-селектор `[id^="yandex_ad"]`. Все добавления
только ДОПОЛНЯЮТ существующие файлы — ни один из уже работавших механизмов не менялся.

**Итого 28 261 правило** — по-прежнему с запасом ниже гарантированного лимита Chrome в
30 000.

## 4. Функции — как они работают технически

- **Бейдж со счётчиком.** `chrome.storage.session` хранит счётчик per-tab (не
  обычная переменная — service worker может быть выгружен браузером в любой
  момент и потеряет обычный массив). Источники счёта: (а) `webRequest.onErrorOccurred`
  ловит `net::ERR_BLOCKED_BY_CLIENT` — то есть браузер уже оборвал запрос по нашему
  DNR-правилу, мы это только наблюдаем, не блокируем сами; (б) `content.js` шлёт
  количество скрытых косметикой элементов.
- **Тумблер «реклама на этом сайте».** При выключении — добавляется динамическое
  DNR allow-правило с приоритетом 2 для `initiatorDomains` этого хоста, плюс
  вкладка перезагружается. `content.js` реагирует «вживую» через
  `chrome.storage.onChanged`, а не через явное сообщение.
  Стелс-режим — это конкретный список известных сервисов (network-level блокировка
  их скриптов), а не универсальный обход детекта. Список требует периодического
  пополнения — сайты меняют детекторы.
- **Свои правила.** Построчно, `domain` — блок, `@@domain` — явное разрешение,
  приоритет 3 (выше списков и вайт-листа сайта).

## 6. Локализация (v1.2.0)

12 языков: ru (запасной по умолчанию), en, zh_CN, es, hi, ar, pt_BR, ja, ko, de, fr, id.
Переводится только то, что видит пользователь: popup, options, название/описание
расширения в chrome://extensions. Внутренний код (комментарии, имена переменных,
ключи сообщений между скриптами) остаётся на английском/русском и не переводится —
пользователь этого не видит.

**Порядок определения активного языка** (в `i18n.js`):
1. Явный выбор пользователя (`chrome.storage.local.languageOverride`) — если есть,
   побеждает всегда.
2. Язык браузера (`chrome.i18n.getUILanguage()`), если он среди 12 поддерживаемых.
3. Русский — запасной вариант, если браузер на языке, которого нет в списке.

**Технически:** `_locales/<код>/messages.json` — формат тот же, что и нативный Chrome
i18n (нужен для `__MSG_extName__`/`__MSG_extDescription__` в manifest.json — это
единственное место, которое обязано следовать языку БРАУЗЕРА, а не выбору пользователя:
имя/описание в chrome://extensions обрабатывается самим Chrome ещё до того, как
запускается наш JS, обойти это невозможно ни для одного расширения). Для всего
остального (popup, options) — свой загрузчик поверх того же формата файлов:
`data-i18n="ключ"` в HTML → `i18n.js` на старте страницы заменяет текст на перевод
активного языка. `data-i18n-html` — то же самое, но через innerHTML (только там, где
внутри перевода нужна разметка вроде `<code>@@domain</code>`, сама разметка одинакова
на всех языках). Арабский — `dir="rtl"`, добавлены точечные CSS-исправления для
тумблеров (flexbox с `row` и так зеркалится в RTL сам по себе, руками правились только
абсолютно спозиционированные элементы).

**Как это проверено на полноту:** `generate_locales.py` при генерации требует перевод
на все 12 языков для каждого ключа — при нехватке скрипт падает, а не тихо оставляет
дыру. Отдельно прогнан скрипт сверки: все ключи `data-i18n*` из HTML/JS ↔ все ключи в
`_locales/ru/messages.json` ↔ идентичность набора ключей во всех 12 файлов — 46 ключей,
полное совпадение, расхождений нет.

Числа (счётчики блокировок) намеренно НЕ локализуются полностью — используется
универсальный формат с суффиксами K/M и обычные западные цифры, а не десятичные
системы счисления конкретного языка (например, деванагари-цифры для хинди). Даты
(«расширение работает с …») локализуются через `Date.toLocaleDateString()` с кодом
активного языка — это уже умеет сам браузер.

- Список — 24 181 правило, не «весь интернет»: длинный хвост мелких/региональных
  рекламных сетей туда не попал (см. RESEARCH.md — там оценка, что можно добавить).
- Косметическая фильтрация — фиксированный список CSS-селекторов, не текстовые
  или процедурные правила (пока не различает, например, div с текстом «Реклама»,
  если у него нет узнаваемого класса/id).
- Никакого JS-уровня — вся защита работает через сетевые правила и CSS. Скрипты
  самих рекламных SDK, если сайт их всё же смог загрузить, не перехватываются и
  не обезвреживаются.
- Один статический слепок списков на момент сборки — обновление требует
  пересборки расширения (это не «живой» список, тянущийся с сервера — так и
  задумано, чтобы не было ни одного внешнего сервера).
