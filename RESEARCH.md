# FlameBlock — исследование: что можно улучшить (v1.1.0)

Ниже — реальный поиск по состоянию блокировщиков рекламы на середину 2026 года
(не по памяти, а с проверкой источников), и по каждой найденной технике —
честный вердикт: применимо ли это к нам на Chrome/Opera, и что я в итоге сделал.

## 1. Контекст рынка на середину 2026 — это меняет расстановку сил

Важная находка, которая прямо касается нашей позиции на рынке: **классический
uBlock Origin на Chrome фактически мёртв**. Google закрывает Manifest V2
полностью в Chrome 150/151 (середина 2026), и на Chrome остаётся только
**uBlock Origin Lite** — сильно урезанная версия того же автора. По независимым
обзорам (PrivacyTools.io, adblock-tester.com, iplocation.net, апрель–июнь 2026):

- uBO Lite **не имеет динамической фильтрации** (правила фиксированы при сборке —
  ровно наша ситуация, только у нас чуть больше правил гарантированно, 24к против
  их меньшего набора)
- uBO Lite **вообще не имеет косметической фильтрации** («drops cosmetic
  filtering... the technique that hides ad containers and page clutter») —
  то есть у нас на СТАРТЕ есть то, чего нет у самого популярного бесплатного
  конкурента на Chrome
- uBO Lite **не имеет scriptlet injection** вообще
- по словам самого автора (Raymond Hill), uBO Lite «struggles against sites that
  actively try to detect and defeat ad blockers» — то есть слабое место как раз
  там, где мы целились сделать стелс-режим

AdGuard считается сейчас лучшим именно extension-решением на Chrome («a
meaningful improvement over Lite, though still short of MV2-era capability»),
но и у него отмечают слабость против агрессивного анти-adblock и на YouTube.
YouTube отдельно называется самым сложным случаем для ЛЮБОГО расширения на
Manifest V3 — это не наша недоработка, а платформенное ограничение, которое
Google сознательно ввёл (та же статья прямо связывает это с рекламной выручкой
Google и оценкой в $239.5 млрд за 2025 год).

**Вывод:** играть предстоит не «с нуля против гигантов», а на одном игровом поле
с ними — все extension-решения на Chrome/Opera сейчас ограничены одним и тем же
API (declarativeNetRequest). Вопрос не «может ли кто-то технически обойти это
ограничение» (нет, у всех те же рамки), а «кто выжимает из этих рамок больше».

## 2. Технику проверил по отдельности

| Техника | Что это | Вердикт для нас | Статус |
|---|---|---|---|
| **MAIN-world script injection** | Внедрение статичного JS прямо в контекст страницы (не в изолированный мир) через `chrome.scripting.registerContentScripts` с `world: "MAIN"` | Официально поддерживается MV3 (Chrome ≥102), подтверждено документацией Chrome for Developers. Требует host permissions — они у нас уже есть. Нельзя грузить удалённый/eval-код (и не нужно — весь скрипт статичный, лежит в архиве) | ✅ Добавил (`scriptlets.js`) |
| **CNAME uncloaking** (раскрытие трекеров, спрятанных за CNAME первого лица) | То, что умеет полный uBO на Firefox | **Технически невозможно на Chrome/Opera** — у Chromium нет `dns`-API для расширений, только у Firefox. Это подтверждает даже собственный инженерный блог Brave (обновлён в феврале 2026): «this solution only works in Firefox, as Chromium does not provide the browser.dns API». Академическая замена есть (ML-классификатор без DNS, статья 2021 года), но это исследовательский прототип, не production-код | ❌ Невозможно в принципе на нашей платформе — не наша недоработка, платформенное ограничение |
| **Процедурные косметические фильтры** (`:has-text()`, `:upward()` и т.п. из uBO) | Скрытие не по CSS-селектору, а по тексту/структуре элемента | Полный движок uBO — это годы отладки под конкретные сайты, повторять один в один нет смысла. Но саму ИДЕЮ (найти по тексту «Реклама»/«Sponsored», скрыть контейнер на N уровней выше) реализовать несложно | ✅ Добавил облегчённую версию в `content.js` |
| **Эвристика по размеру iframe** (баннеры стандартных рекламных размеров 300×250, 728×90 и т.д.) | Ловит рекламу, которую сайт показывает через собственный (first-party) прокси-домен — обычные списки такое не видят в принципе, потому что домен «свой» | Работает, но выше риск ложных срабатываний, чем у списков — не любой iframe 300×250 обязательно реклама | ✅ Добавил, но **выключено по умолчанию**, включается отдельным тумблером в настройках |
| **Гейт на `window.open`** (защита от попандеров/принудительных редиректов) | Блокирует вызовы `window.open`, не привязанные к настоящему клику пользователя | Дешёвая, надёжная, малый риск что-то сломать — довольно стандартный приём | ✅ Добавил в `scriptlets.js` |
| **CSP-инъекция через `modifyHeaders`** | Переписывать Content-Security-Policy ответа, чтобы жёстче резать скрипты | DNR такое умеет (`action.type: "modifyHeaders"`), но по багтрекеру Chromium (issue 40825809) поведение подтверждённо нестабильно в части случаев, и это не точечная штука — переписывание CSP всего домена легко ломает не только рекламу, но и саму работу сайта | ⏸ Отложил — риск поломки сайтов выше пользы, нужно тестировать точечно, не блочить |
| **Расширение списка ещё сильнее (AdGuard Base filter и т.п.)** | Добавить ещё один crowd-sourced список поверх EasyList/EasyPrivacy | Технически просто, есть запас (использовано 24к из гарантированных 30к) | 📋 Осознанно отложил на следующий заход — это количественный рост, а не новое качество, приоритетнее было добавить то, чего вообще не было (см. выше) |

## 3. Что добавлено в v1.1.0 конкретно

1. **`scriptlets.js`** — статический скрипт, встраивается в MAIN world каждой
   страницы через `chrome.scripting.registerContentScripts` (регистрируется из
   `background.js`, включается/выключается тумблером стелс-режима).
   - Гейт на `window.open`: пропускает вызов, только если он произошёл в течение
     ~1 секунды после реального клика/нажатия клавиши пользователем — типичный
     попандер-скрипт этого не пройдёт.
   - Мост обратно в счётчик: событие `CustomEvent` в DOM, которое подхватывает
     `content.js` (MAIN world не имеет доступа к `chrome.*` API — это ограничение
     самой платформы, не наше).
2. **Текстовая эвристика в `content.js`** — ищет короткие текстовые метки
   («Реклама», «Sponsored», «Advertisement», «Promoted», «На правах рекламы») и
   прячет ближайший разумный контейнер-обёртку, а не только сам текст.
3. **Эвристика по размеру iframe** — отдельный, по умолчанию выключенный слой:
   iframe стандартных рекламных размеров с чужого домена. Осознанно не включил
   по умолчанию — ложные срабатывания возможны, пусть решает пользователь
   (тумблер «Эвристика по iframe» в настройках, помечен как экспериментальный).

## 4. Честно — что осталось за скобками и почему

- Полноценный движок скриптлетов «как в классическом uBO» (с параметрами,
  привязкой к конкретным сайтам, десятками готовых обезвреживателей) — это
  отдельный, годами обкатываемый проект, а не то, что можно безопасно добавить
  за один заход без риска сломать половину сайтов.
- CNAME uncloaking — не вопрос усилий, а platform-ограничение Chromium. Появится
  только если Google добавит DNS API для расширений (пока нет никаких признаков).
- Идеальная защита от YouTube-специфичных антиблокировщиков — по независимым
  тестам этого не может дать вообще ни одно Chrome/Opera-расширение сейчас,
  включая AdGuard и uBO Lite; тут разница только в системных приложениях уровня
  ОС, которые действуют не как расширение браузера, а совсем другим способом.
